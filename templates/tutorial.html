What is a Formal Grammar?

In the Theory of Automata and Formal Languages (TAFL), a grammar is a mathematical system that defines a set of strings (a language). It provides a precise mechanism for generating every valid string in the language, and rejecting everything else.

Formal Definition

A grammar is defined as a 4-tuple:

Grammar G = (V, T, P, S)
  • V = Set of variables (non-terminals). These are placeholders like A, B, C, S that represent abstract categories. They act as intermediate "building blocks" that get replaced during derivation.
  • T = Set of terminals. These are the actual symbols that appear in the final output string: a, b, c, 0, 1, +, *, (, ), etc. They cannot be broken down further.
  • P = Set of production rules of the form α ⇒ β, where α contains at least one variable, and β is any combination of variables and terminals. These are the "recipes" that tell you how to rewrite symbols.
  • S = The start symbol (S ∈ V). Every derivation must begin from this special non-terminal.
Key Insight: The constraints placed on the production rules determine the type of grammar. Weaker constraints allow more expressive languages but require more powerful machines to recognize them. This is the foundation of the Chomsky Hierarchy.

The Chomsky Hierarchy

Noam Chomsky classified grammars into four types based on the form of their production rules:

Type Name Production Rule Form Recognizer
Type 3 Regular Grammar A → wB or A → w Finite Automaton (DFA/NFA)
Type 2 Context-Free Grammar A → α, |A| = 1 Pushdown Automaton (PDA)
Type 1 Context-Sensitive Grammar αAβ → αγβ, |γ| ≥ 1 Linear Bounded Automaton
Type 0 Unrestricted Grammar α → β (no restrictions) Turing Machine
This tool focuses on Type 2: Context-Free Grammars (CFGs), which are powerful enough to describe programming language syntax, mathematical expressions, and nested structures, while remaining efficiently parsable.

Why Do We Need Grammars?

The motivation behind formal grammars is deeply practical:

  • Compiler Design: Every programming language (Python, C++, Java) has its syntax defined by a CFG. When you write code and the compiler tells you "syntax error", it's because your code failed to derive from the language's grammar.
  • Natural Language Processing: Understanding sentence structures in English, Hindi, or any natural language relies on grammar parsing.
  • Data Validation: XML, JSON, HTML, all structured data formats are formally defined by grammars.
  • Pattern Matching: Regular expressions are a direct application of Regular Grammars (Type 3).

Type 3 Regular Grammar (RG)

A Regular Grammar is the simplest type in the Chomsky hierarchy. It generates exactly the set of regular languages, the same languages recognized by Finite Automata (DFA/NFA) and described by Regular Expressions.

Two Forms of Regular Grammar

Regular grammars come in two flavors, and a grammar must be purely one or the other (never mixed) to qualify as regular:

Right Linear Grammar
A → wB
A → w

The non-terminal appears on the right end.
w ∈ T* (string of terminals)
Left Linear Grammar
A → Bw
A → w

The non-terminal appears on the left end.
w ∈ T* (string of terminals)

What Makes a Grammar NOT Regular?

A production like A → aAB or A → AB is not regular because it has non-terminals on both sides of the replacement, or multiple non-terminals appearing together. This violates the strict linear constraint.

Examples A → ab A     ✅ Right Linear Regular Grammar
A → A ab     ✅ Left Linear Regular Grammar
A → a B a    ❌ NOT Regular (non-terminal in the middle)
A → AB       ❌ NOT Regular (two non-terminals on RHS)

Writing Grammars for Regular Languages

Given a regular expression, we can systematically write a right-linear grammar:

Language: (a+b)*(aa+bb)
S → aS | bS | aa | bb
Language: (a+b)*(ab+ba)
S → aS | bS | ab | ba

Conversion: DFA to Regular Grammar (RG)

Every DFA can be systematically converted into a right-linear regular grammar. The process maps states to non-terminals and transitions to production rules:

Conversion Rules:
Given a DFA with transition function δ(A, a) = B:

1 If B is not a final state: write A → aB

2 If B is a final state: write A → aB | a (add both the recursive and terminating form)

Regular Grammars are a strict subset of CFGs. Every regular grammar is also context-free, but not every CFG is regular. Regular grammars cannot handle nested/recursive structures like balanced parentheses or anbn.

Type 2 Context-Free Grammar (CFG)

A Context-Free Grammar is more powerful than a regular grammar. The key difference: in a CFG, the left-hand side of every production rule is a single non-terminal, and the right-hand side can be any combination of terminals and non-terminals.

CFG Production Rule Form A → α     where |A| = 1, A ∈ V, and α ∈ (V ∪ T)*

The constraint is minimal: the left-hand side must be exactly one non-terminal (hence "context-free", since the replacement happens regardless of surrounding context). The right side α can be any string of terminals and/or non-terminals, including ε.

Why CFGs? The Motivation

Regular grammars (and finite automata) have a fundamental limitation: they have no memory. They can't count or match patterns. Context-Free Languages overcome this because:

Behind the scenes, CFLs use a stack (memory). A Pushdown Automaton (PDA) recognizes CFLs by pushing and popping symbols on a stack. If a language can be recognized using a stack, it is context-free.

Classic CFG Examples

L₁ = {aibjck | i, j ≥ 1}
✅ Context-Free

S → AB
A → aA | a
B → bB | b

No relationship between counts of a, b, c needed. Each group is generated independently.
L = {aa⁺b⁺} (one or more a's, then b's)
✅ Context-Free

S → AB
A → aA | a
B → bB | b

The counts of a's and b's are independent, both just need ≥ 1.

Languages at the Boundary

L₁ = {aibjck | i, j ≥ 1}
✅ Context-Free

S → AB
A → aA | a
B → bBc | bc

The stack can match b's with c's (push b, pop on c). a's are independent.
L₂ = {aibicj | i, j ≥ 1}
✅ Context-Free

S → AB
A → aAb | ab
B → cB | c

The stack matches a's with b's. c's are generated independently.
L₃ = {aibici | i ≥ 1} is NOT Context-Free!
This requires matching three counts simultaneously. A stack can push a's and pop on b's, but then it can't also ensure equal c's. You need a Turing machine (or at minimum a Context-Sensitive Grammar) for this.

CFG for Palindromes

One of the most elegant CFG applications is generating palindromes over {a, b}:

Odd-Length Palindromes: L = {w | w is an odd-length palindrome} S → aSa | bSb | a | b

The grammar wraps matching symbols on both sides (using the stack to "remember" the first half), then places a single character in the center to ensure odd length.

Derivations: How Strings Are Generated

A derivation is a sequence of rule applications that transforms the start symbol into a string of terminals. There are two canonical orders:

Leftmost Derivation (LMD)
At each step, always expand the leftmost non-terminal first. This is deterministic: given a parse tree, there is exactly one LMD.
Rightmost Derivation (RMD)
At each step, always expand the rightmost non-terminal first. This is the order used by bottom-up parsers like LR parsers (in reverse).

Ambiguity

A grammar is ambiguous if there exists any string that has two or more distinct leftmost derivations (equivalently, two or more distinct parse trees). The classic example:

Ambiguous Grammar E → E + E | E * E | id

For the string id + id * id, this grammar produces two different parse trees: one that evaluates + first, and another that evaluates * first. This ambiguity is a real problem in compilers because it leads to different computation results.

This tool detects ambiguity automatically! The Ambiguity Checker generates strings from the grammar and tests each for multiple parse trees, providing a visual proof if ambiguity is found.

Under the Hood: How Every Function Works

This section explains the architectural components of the visualizer. The backend is written in Python (Flask), and the frontend uses vanilla JavaScript with SVG rendering for parse trees.

Backend: Parsing Pipeline

When you click Derive, the following pipeline executes:

parse_grammar(raw_text)
Purpose: Converts raw text from the grammar input box into a structured dictionary.
How: Splits input by lines, replaces -> and ::= with →, separates alternatives by |, and normalizes epsilon variants (ε, epsilon, eps) into a canonical form. Returns the grammar dictionary, start symbol, set of non-terminals, and any error message.
Motivation: Users type grammars in many notations. This function provides a robust, forgiving parser that accepts any common format.
earley_parse(grammar, start, tokens, non_terminals)
Purpose: The core recognition engine. Determines whether the input string can be derived from the grammar, and if so, extracts a complete parse tree.
How: Implements the Earley Parsing Algorithm, which operates in O(n³) time for general CFGs. It maintains a "chart" of Earley items (partially completed rules), advancing through three operations: Prediction (introduce new rules for non-terminals), Scanning (match a terminal from input), and Completion (propagate completed rules back).
Motivation: Unlike naive BFS derivation which can loop infinitely on recursive grammars like S → S S, the Earley algorithm always terminates and handles any CFG, including ambiguous, left-recursive, and right-recursive grammars.
earley_parse_multi(grammar, start, tokens, non_terminals, max_trees)
Purpose: A modified version of the Earley parser that finds multiple distinct parse trees for the same input string.
How: Instead of storing one item per chart key, it stores up to max_trees items with structurally different trees. Uses _trees_structurally_equal() to detect duplicates.
Motivation: Ambiguity detection requires proving that two different parse trees exist for the same string. This function is the engine behind the Ambiguity Checker.
extract_derivation(tree, non_terminals, leftmost=True)
Purpose: Given a parse tree, produces the step-by-step derivation sequence (either leftmost or rightmost).
How: Walks through the tree, maintaining a "sentential form" (mixed sequence of terminals and non-terminal subtrees). At each step, it locates the leftmost (or rightmost) non-terminal subtree, replaces it with its children, and records the new form with highlight information.
Motivation: The derivation steps are the core educational output. They show students exactly how each rule is applied to transform the start symbol into the final string.
build_tree_snapshots(tree, non_terminals, leftmost=True)
Purpose: Creates incremental snapshots of the parse tree growing step-by-step.
How: Collects the expansion order of non-terminal nodes (leftmost-first or rightmost-first), then for each step, builds a partial tree showing only the nodes expanded so far.
Motivation: Powers the animated parse tree visualization. Each snapshot corresponds to a derivation step, so the tree grows in sync with the step-by-step derivation.
generate_derivation_explanation(steps_raw, grammar, non_terminals, leftmost)
Purpose: Generates rich, human-readable explanations for each derivation step.
How: Analyzes the transition between consecutive steps: identifies which non-terminal was expanded, which production rule was used, what new terminals and non-terminals appeared, and constructs a natural language explanation.
Motivation: Raw derivation steps can be cryptic for beginners. These explanations turn each step into a teaching moment, explaining why a particular non-terminal was chosen and what effect the rule application has.
generate_strings(grammar, start, non_terminals, ...)
Purpose: Generates all terminal strings derivable from the grammar up to a given length, using BFS over leftmost derivation.
How: Starting from the start symbol, it iteratively applies all possible productions to the leftmost non-terminal, collecting complete strings (those with no remaining non-terminals) and pruning forms that exceed the length limit.
Motivation: The Ambiguity Checker needs test strings to check. This function systematically enumerates the language of the grammar, providing candidates for ambiguity testing.
check_grammar_ambiguity(grammar, start, non_terminals, max_length)
Purpose: The top-level ambiguity detection function.
How: Generates strings using generate_strings(), then tests each using earley_parse_multi() to find strings with 2+ parse trees. If found, it extracts both derivations and computes the divergence point.
Motivation: Manually checking for ambiguity is tedious and error-prone. This automates the process and provides a visual witness (two distinct derivations) as proof.

Frontend: Rendering Pipeline

buildStepDom(steps, container, noneEl, explanations, mode)
Purpose: Renders derivation steps as interactive, color-coded DOM elements.
How: For each step, creates a row with a step number, colored tokens (orange for non-terminals, green for terminals, purple underline for the NT being expanded next), and an expandable explanation panel.
Color coding: Non-terminal | Terminal | Next to expand
renderTree(tree, svgEl, markerId)
Purpose: Renders a parse tree as an SVG graphic with animated nodes, edges, and arrow markers.
How: Recursively computes node widths and positions using a top-down layout algorithm, then draws circles for nodes and lines for edges. Nodes pop in with a spring animation via CSS transitions.
Node types: Orange border = non-terminal | Green border = terminal | Italic orange = ε (epsilon)
Smart Auto-Formatting (formatGrammarInput / formatStringInput)
Purpose: Automatically spaces every symbol as you type, converts -> to →, and prevents manual spacebar input.
How: Strips all whitespace from the raw input, then re-inserts a single space after every character (preserving newlines). For the string input, it uses greedy longest-match tokenization against the grammar's terminal symbols.
Motivation: Consistent formatting eliminates tokenization errors and makes grammars more readable. The spacebar lock ensures users can't introduce ambiguous spacing.

Restrictions, Limits & Troubleshooting

Performance Cap: For optimal performance without freezing your browser, please keep target input strings to a maximum of 50 to 70 symbols. The Earley parsing algorithm runs in O(n³) time, mathematically guaranteeing accuracy, but extreme lengths (100+ tokens) will significantly slow down.

System Guarantees

  • Infinite Loop Protection: This system uses the Earley Parser. This guarantees the application will never infinitely loop or crash, even if you define infinite recursive loops in your grammar like S → S S. The parsing will always successfully halt and resolve.
  • Mandatory Halting: The system will intentionally halt processes and declare failure strictly when:
    • The typed string uses symbols that don't exist in your grammar vocabulary.
    • The structural derivation is fundamentally impossible under the defined rules.
  • Auto-Formatting Lock: Manual "Spacebar" input is strictly ignored! Do not fight the formatter. The internal logic handles all token extraction flawlessly.

Decoding Common Errors

"Input symbol is a non-terminal"
You entered an incomplete sentence. The string input box represents final output words only (terminals). You cannot test symbols that represent an abstract rule structure. For example, if your grammar has S → AB, you cannot type A or B as input; you must type the final terminal symbols like a b.
"Derivation not possible"
Don't panic! It's not a bug. This is the correct mathematical response whenever the specific string you supplied cannot be generated by applying the grammar rules. Double-check:
  • Are your grammar rules correct?
  • Does the string use only symbols defined as terminals in your grammar?
  • Is the string actually in the language defined by the grammar?
"Invalid rule (missing →)"
Each line of your grammar must contain an arrow symbol. Use the → button or type -> which will be auto-converted. Every rule must be in the form NonTerminal → body.
